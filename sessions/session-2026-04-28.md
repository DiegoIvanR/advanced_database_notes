## What I understood
- schema backup in this context means extracting DDL (structure), not actual data
- DBMS_METADATA.GET_DDL allows us to reconstruct objects like tables, views, indexes, and code
- transform parameters help make DDL cleaner and portable by removing storage, tablespace, and schema-specific details
- setting EMIT_SCHEMA = false is important when migrating to a different schema
- unlike expdp, this method does not export data and requires manual handling
- dependencies between objects matter, so the order of creation is critical during restore
- tables must be created before constraints, and before objects like views or procedures that depend on them
- foreign keys and schema-qualified references can break during migration and must be reviewed
- USER_DEPENDENCIES helps identify which objects rely on others
